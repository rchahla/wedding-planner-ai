1. The Problem

  • Wedding planning involves juggling dozens of interdependent decisions — venue, budget, catering, weather, guest count
  • Couples often discover conflicts too late (e.g., budget won't cover the guest list, outdoor venue in winter)
  • Generic online advice doesn't account for your specific numbers
  • Professional planners cost $2,000–$5,000+

Our Solution

An AI assistant that takes your event details, checks them against a curated wedding planning knowledge base, flags real conflicts with clarifying questions, and generates a personalized planning report — complete with source citations, actionable next steps, weather risk assessment, and email delivery.

Key value: Catch planning conflicts early, get advice grounded in real planning data, not LLM hallucinations.

2. System Architecture

RAG Pipeline

10 domain documents (budget, catering, venue, outdoor, photography, entertainment, decor, timeline, vendors, guests)

Embeddings: all-MiniLM-L6-v2

Vector Store: ChromaDB (persistent, incremental ingestion)

Retrieval: Top-3 by cosine similarity

State & Workflow

State Tracking: 6 event fields persisted to JSON across all pipeline steps

Workflow: 6-step agentic pipeline with conditional branching

Branching: Conflicts halt pipeline; user must resolve or acknowledge

External Tools

Gemini 2.5 Flash: Conflict detection, report generation, follow-up chat

OpenWeatherMap: Live weather for outdoor venue risk

SendGrid: HTML email report delivery

Chart.js: Budget visualization

Stack: Flask + Vanilla JS | ChromaDB + SentenceTransformers | Gemini API | OpenWeatherMap | SendGrid

Conflict Detection (LLM-Grounded)

Gemini evaluates event details against retrieved documents. Only flags hard conflicts — numeric violations backed by specific source documents.

Fallback: 4 rule-based checks if LLM fails (budget threshold, missing date, seasonal risk, per-head catering cost).

Over-flagging guard: >4 conflicts triggers fallback to prevent hallucinated issues.

Conditional Branching

If conflicts exist and are not acknowledged, the pipeline halts at Step 4 and returns a clarification request.

The user must either fix their inputs or check the "acknowledge" box to proceed.

This prevents generating a report the user can't trust.

How Grounding Works

  • Composite query built from all 6 state fields for better recall
  • Retrieved docs injected into both conflict and report prompts with source tags
  • LLM instructed to cite *(Source: filename.txt)* after each recommendation
  • Follow-up chat gets full event state + report as context

Example: 120 guests, $5K budget → retrieves budget_guide.txt → Gemini flags: "According to budget_guide.txt, $8,000-$15,000 is typical for 80+ guests"

5. Structured Output Artifact

Report Structure

1. Event Summary

Restates key event details (guest count, budget, theme, date, venue, dietary)

2. Conflict Summary

Explains detected conflicts and their planning implications

3. Planning Recommendations

4-6 specific recommendations, each citing the source document

4. Suggested Next Actions

3-5 concrete checklist items the couple should take

Output Features

  • Markdown format — rendered in the UI with marked.js, saved to outputs/wedding_report.md
  • Source citations — every recommendation cites a specific knowledge base document
  • Budget doughnut chart — visual breakdown of allocation (Chart.js)
  • Email delivery — HTML-styled report sent via SendGrid
  • Follow-up Q&A — chat interface grounded in the generated report
  • Fallback template — deterministic report if Gemini is unavailable

6. Reliability & Fallbacks

Graceful Degradation

  • No Gemini API → rule-based conflict detection + static report
  • No Weather API → weather section omitted
  • No SendGrid → email step skipped
  • LLM over-flags → fallback to deterministic rules

Prompt Engineering

  • "Hard conflicts only" — explicit negative instructions reduce false positives
  • NO_CONFLICTS sentinel for deterministic parsing
  • Over-flagging guard (>4 conflicts)
  • Structured output format enforced via prompt

Known Limitations

  • Single-city weather (London, ON)
  • No multi-user sessions
  • Static budget chart percentages
  • 10-document corpus scope

Design principle: Every external dependency has a fallback. The system always produces useful output, even if degraded.
